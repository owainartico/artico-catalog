## Critical Fixes Applied (Evening - Opus Review)

**Status:** Opus agent reviewed entire codebase and conversation. Identified 4 critical bugs:

### 1. Auth Middleware Bug (ROOT CAUSE of "Unexpected token" error)
- **Problem:** eq.path is relative to mount point, so eq.path.startsWith('/api/') never matched for API routes
- **Result:** API requests got HTML login page instead of JSON 401 ? "Unexpected token" when parsing
- **Fix:** Changed to eq.originalUrl.startsWith('/api/')

### 2. Items Route Missing Status Filter
- **Problem:** Items endpoint didn't filter status = 'active', only filtered lifecycle_status
- **Result:** Inactive items like RFS11363 showed in forecast page
- **Fix:** Added i.status = 'active' filter by default (unless include_inactive param)

### 3. No Defensive JSON Parsing
- **Problem:** Frontend pi() function didn't handle HTML responses
- **Result:** When auth failed silently, got HTML ? JSON.parse() threw error
- **Fix:** Added try/catch around JSON.parse, detect HTML, redirect to login

### 4. Composite Sync Too Slow
- **Problem:** Checked ALL 4,000 items individually (1 API call each = 200+ seconds)
- **Result:** Render timeout, composite sync failing
- **Fix:** Only check items where is_composite = true (much faster)

## Incremental Sales Sync (Evening)

**Problem:** Sales sync was re-syncing 365 days of invoices every time ? timeout on Render, slow, wasteful

**Solution (commit 72ff492):**
- Sales sync now **incremental by default** ? only fetches invoices since last successful sync
- First sync: Gets full 365 days to build history
- Subsequent syncs: Only gets new invoices (much faster, ~7-30 days max)
- Added ?full=true option for complete 365-day resync when needed

**How it works:**
1. Checks sync_log table for last successful sales sync date
2. If found: Only syncs from that date forward (+ 7 day buffer for delayed invoices)
3. If not found (first sync): Gets full 365 days
4. Pass ?full=true to force complete resync

**API:**
- POST /api/sync/sales ? Incremental (fast, only new invoices)
- POST /api/sync/sales?full=true ? Full resync (365 days)
- POST /api/sync/all ? Incremental
- POST /api/sync/all?full=true ? Full resync

**Benefits:**
- Much faster regular syncs (~30 seconds vs minutes)
- No timeouts
- Less API calls to Zoho
- Still catches all data (checks 7 days back as buffer)

## Critical Sales Sync Fixes (Afternoon)

**Problem:** Sales sync only found 3 invoices when user had "way more than 3 invoices in last 7 days"

**Root causes found:**

1. **Only fetching status=paid invoices**
   - Code had: /v3/invoices?status=paid
   - Missing: "sent", "issued", "partially_paid", "draft" invoices
   - Fix: Removed status filter ? fetches ALL invoices

2. **Skipping entire invoices if ANY line item existed**
   - Logic checked: "Does this invoice exist in sales_history?"
   - If yes: Skipped ENTIRE invoice (all line items)
   - Problem: Invoice with 10 line items ? if 1 synced, skipped all 10
   - Fix: Removed invoice-level check, rely on line-item-level unique constraint

**Commit:** bcea36b

**How it works now:**
- Fetches ALL invoices (any status: sent, issued, paid, partial, etc.)
- Processes every line item
- Unique constraint (zoho_invoice_id, item_id) prevents true duplicates
- ON CONFLICT DO NOTHING skips only actual duplicate line items

**Expected result:**
- Full sales resync should now fetch hundreds/thousands of invoices
- All invoice statuses included (not just paid)
